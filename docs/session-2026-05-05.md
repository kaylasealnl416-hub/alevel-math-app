### 反转 1：以为 math_ 前缀是新版正确的

→ 调查后发现：**新版 math_ 全部空壳，旧版纯前缀有 273 道题 + 全部内容**。推荐"删 math_ 空壳"。

### 反转 2：以为整个项目共识用 math_ 前缀

→ grep 全库后发现：**只有 seed-mathematics.js 一个文件用 math_ 前缀**，前端 `curriculum.js` / 后端 `topicChapterMap.js` / `data-import/subjects.js` / wiki / 273 道实际题目**全部用旧版纯前缀**。

3 月 18 日 commit `4a25e1f` "Add mathematics subject seed script (... math_ prefixed IDs)" 是 Sonnet 4.6 协作时的**单方面提议**——只创建了 47 个空壳骨架，没改其他任何代码，从未被项目接受。

### 反转 3：发现 math_ 前缀的真正动机

调查发现 `p1c1` 等 11 个 ID 在 DB 里实际归属是 **politics**（UK Democracy 等）：
- `p1c1` = politics Unit1 Ch1 (22 道题)
- `p1c2` = politics Unit1 Ch2 (15 道题)
- `p2c1`-`p2c3` = politics Unit2 (41 道题)
- `p3c1`-`p3c3` = politics Unit3 (3 道题)

**真相浮出水面**：项目根 CLAUDE.md 第 7 行表格说"Politics 前缀已修复 p → pol"，但 DB 从未真正迁移——politics 仍然占着 `p_` 前缀，正好跟 mathematics 的 P1 Ch1 等冲突。Sonnet 4.6 看到这个冲突，给 mathematics 加 `math_` 前缀绕开。**他的诊断是对的，但只补了 mathematics 这一头，没修 politics**。8 周烂摊子完整保留。

## 四、第三阶段：5 阶段数据卫生整改

### 阶段 1：politics `p_` → `pol_` 迁移（事务保护）

`backend/scripts/db-cleanup/migrate-politics-prefix.mjs --execute`

由于 `questions.chapter_id` 等 FK 没有 `ON UPDATE CASCADE`，必须按以下顺序在事务里：
1. INSERT 新 `pol_` chapter（复制旧 chapter 全部字段，仅改 id 和 createdAt）
2. UPDATE 子表 FK：questions / learning_progress / question_sets
3. DELETE 旧 `p_` chapter

**结果**：87 questions + 1 progress + 4 sets + 11 chapters，全部成功迁移。
`user_answers` (18) / `exam_question_results` (31) 通过 `question.id` 关联，未受影响。

### 阶段 2-4：mathematics 清理

`backend/scripts/db-cleanup/drop-math-prefix-orphans.mjs --execute`

- 36 条 `math_*` 重复对空壳 → DELETE
- 11 条孤立 `math_*`（math_p1c1 等）→ RENAME 为 `p1c1` 等（politics 让位后 ID 已空闲）
- ALTER TABLE chapters ADD CONSTRAINT `chapters_unit_num_unique UNIQUE (unit_id, num)`

**结果**：mathematics 47 唯一章节、0 重复，约束生效防复发。

## 六、教训（避免下次重蹈覆辙）

1. **判断"项目设计意图"前必须扫全局采纳情况**——单 commit message + 单 file 实现 ≠ 项目接受
2. **DB 迁移涉及 FK 时必须用 INSERT-then-DELETE 模式**——直接 UPDATE PK 会被 FK 约束拦截
3. **数据完整性约束应该在 schema 设计时就加**——chapters 表如果一开始就有 `(unit_id, num)` 唯一约束，3 月 18 日的半截迁移会立即被 DB 拒绝，根本不会留 8 周
4. **dry-run 必须充分预检**——这次三次反转都是因为 dry-run 暴露了之前没意识到的现实，每一次都救了一把
5. **ADR / 决策记录必须及时落地**——CLAUDE.md 上层说"politics 已修复 p → pol"，但 DB 从未对齐，导致后续推断全部错位
